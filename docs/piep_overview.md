# PIEP Overview

## What PIEP is

PIEP is an interactive crystallographic analysis program for determining a 3D unit cell from multiple diffraction patterns recorded from randomly oriented crystals, and then indexing individual patterns against that cell. The codebase is primarily FORTRAN 77 and was built around electron diffraction use cases, but the underlying geometry is more general: each pattern contributes two experimentally measured reciprocal-space basis vectors and the program searches for a consistent 3D reciprocal lattice.

The recently published paper describes the core search method as the GM algorithm. The source code shows that PIEP is broader than that single algorithm: it also contains pattern preparation, memory management, cell transformation and reduction tools, indexing support, file I/O helpers, and command-driven workspaces for exploratory analysis.

## Core capabilities

PIEP's practical scope is centered on four tasks:

1. Import and store multiple diffraction-pattern descriptions, each consisting of two reciprocal basis vectors and their mutual angle.
2. Search for candidate 3D unit cells that are compatible with a selected subset of those patterns.
3. Reduce, transform, compare, and store candidate cells in multiple in-memory banks.
4. Index individual patterns by assigning Miller indices to the two measured basis vectors and a zone axis to the pattern.

In the interactive workflow this appears through commands such as `PG`, `AP`, `PC`, `DC`, `DE`, `MV`, `MA`, `CW`, and `I`.

## Data model and operating model

PIEP behaves like an interactive workstation rather than a single-purpose batch executable.

- A-memory stores diffraction-pattern descriptions.
- B-memory stores manually entered or transformed cell descriptions.
- C-memory stores cell candidates generated by the `DC` search.
- Parameter and log files define defaults, output destinations, and formatting conventions.

The `scheme1.png` flowchart is especially important because it makes clear that the program is not just "run search, print result". It is a stateful environment in which the user iteratively loads patterns, chooses search settings, runs cell determination, transforms cells, checks conventional settings, and performs indexing.

## Scientific use case

The intended use case is sparse, heterogeneous diffraction data where:

- each pattern yields only a local 2D reciprocal basis,
- different patterns come from unrelated crystal orientations,
- some patterns may be poor, distorted, or alien,
- the user can supply approximate experimental tolerances and a plausible volume range.

This is exactly the niche where exhaustive or semi-exhaustive reciprocal-space search can be preferable to more assumption-heavy methods.

## Strengths

The legacy implementation has several strong properties that should be preserved in a rewrite.

- It is narrow and scientifically purposeful: the code is optimized around unit-cell determination from randomly oriented patterns, not around generic crystallographic bookkeeping.
- It contains explicit symmetry-aware search modes that can reduce a 3D search to 2D or 1D when the reference pattern suggests higher metric symmetry.
- It integrates search and indexing: candidate cells are evaluated by how well they can index the measured pattern vectors.
- It already encodes a great deal of domain knowledge about acceptable tolerances, reduced cells, symmetry restrictions, and near-duplicate candidate handling.

## Limitations

The paper, supplementary examples, and source all show clear limitations.

- The search is only as good as the extracted pattern vectors and their error estimates.
- The main search depends strongly on the chosen reference pattern and inferred symmetry class.
- Overestimating symmetry can exclude the true solution from the search space.
- Underestimating the required volume range can miss the correct cell entirely.
- Runtime can grow sharply when the search falls back to the general 3D or broad 2D cases.
- The interactive architecture makes behavior reproducibility and automated regression testing harder than in a modern library design.

There is also a structural limitation: much of the codebase mixes UI, shared mutable state, scientific logic, and formatting, which makes maintenance difficult even when the underlying numerical method is sound.

## What should be preserved in a rewrite

The rewrite boundary should be defined around behavior, not around the old command set.

The scientific kernel that must be reproduced exactly is:

- reciprocal/direct-space geometry conversions,
- preparation of admissible pattern windows from measurement tolerances,
- GM candidate-cell generation from a chosen reference pattern,
- candidate evaluation through pattern indexing,
- candidate ranking and near-duplicate suppression,
- cell reduction and conventionalization needed for comparing outputs to legacy runs.

The interactive command shell, flat file conventions, and memory-bank UX are historically useful, but they are not the essential product.

## Recommended rewrite target

The modern replacement should be a layered system:

- a deterministic C++ core library for geometry, search, indexing, and cell transformations,
- Python bindings for scripting, notebooks, and integration into modern data-processing pipelines,
- a thin compatibility CLI for reproducing legacy workflows and for regression testing against the published examples.

That split preserves the scientific value of PIEP while removing the main maintenance burden of the legacy architecture.
